# plan-executor

Chinese documentation: [README.zh-CN.md](./README.zh-CN.md).

`plan-executor` is an Agent Skill for implementing reviewed plans. It can execute a user-provided plan, a source plan document generated by `plan-doc-maker`, or the current project's default `docs/plans/plan_index.md`.

The skill reads each active modification point, checks the current code before editing, implements only missing work, reviews each change, updates plan status, verifies the result, then commits and pushes when all active items are complete.

## Install

From the repository root:

```bash
node install.mjs plan-executor
```

From this skill directory, the local wrapper forwards to the root installer:

```bash
node install.mjs
python3 install.py
```

Useful installer options:

```bash
node ../install.mjs plan-executor --dry-run
node ../install.mjs plan-executor --only codex
node ../install.mjs plan-executor --only cursor
node ../install.mjs plan-executor --only claude
node ../install.mjs plan-executor --copy
```

## Usage

Default plan execution:

```text
Use $plan-executor to execute the pending items in docs/plans/plan_index.md, review each change, update statuses, then commit and push.
```

Execute a specific source plan:

```text
Use $plan-executor to implement docs/plans/20260605-1200-api-refactor.md, update its execution status, then commit and push.
```

## 工作流程

`plan-executor` 的核心流程如下：

1. 定位执行来源：用户方案、源方案文档，或默认 `docs/plans/plan_index.md`。
2. 执行前检查当前项目和所有目标项目的 git 工作区；如果存在未提交文件，先分析 staged、unstaged 和 untracked 内容，判断每个文件是否应提交。
3. 对需要保留的既有变更，按所解决的问题、修改类型和意图分批提交，例如文档、实现、测试、工具、生成产物或格式化变更。
4. 重复检查和分批提交，直到所有应该先提交的既有变更都已提交；剩余未提交内容必须明确原因且可安全共存。
5. 拆分待执行修改点或临时执行清单。
6. 对每个修改点先阅读当前代码，判断是否已全部实现、部分实现或未实现。
7. 如果代码已经全部满足方案要求，不改代码，只做验证并维护方案状态为已执行。
8. 如果代码是部分实现或未实现，按方案继续实施未完成内容。
9. 每完成一个修改点，执行评审、修正、再评审，直到通过。
10. 更新 `plan_index.md`、源方案文档或临时执行清单状态。
11. 重新计算整体执行状态，确认没有遗漏。
12. 所有活跃修改点完成后，提交并推送。
13. 结束前输出本次执行汇总报告，说明执行来源、前置提交、完成项、代码改动或无需改代码的原因、方案状态更新、验证结果和提交推送结果。

## 状态维护

当执行来源包含 `docs/plans/plan_index.md` 时，skill 会维护每个活跃修改点的：

- `评审状态`
- `执行状态`
- `状态记录`
- 验证证据或验收结果

当修改点引用 `plan-doc-maker` 生成的源方案文档，或用户直接提供源方案文档时，skill 会同步维护源方案中的实现状态，例如：

- `代码实现: 部分执行 | 已执行`
- `实现文档版本`
- `修改时间`
- 实现记录或等价状态字段

只有所有活跃修改点都执行并评审通过后，skill 才允许进入最终提交推送步骤。

## 汇总报告

每次执行结束前，`plan-executor` 都需要向用户输出本次执行汇总报告。报告至少说明：

- 本次执行的方案来源。
- 完成了哪些修改点或临时清单项。
- 实际做了哪些代码改动，或说明代码已满足方案所以未改代码。
- 执行前发现了哪些既有未提交变更、哪些已分批提交及 commit hash、哪些保留未提交及原因。
- 更新了哪些方案状态字段。
- 做了哪些验证，哪些验证未执行及原因。
- commit 与 push 结果。

## Workflow

1. Locate the execution source.
2. Build the active modification-point set.
3. Inspect current code before editing.
4. Implement missing or partial work.
5. Run a review-fix loop for each item.
6. Update `plan_index.md` and source plan statuses.
7. Recompute overall status.
8. Verify proportionally to the touched surface.
9. Commit, push, and report the result.

## Files

- `SKILL.md`: trigger rules, execution loop, verification rules, status maintenance, and final response contract.
- `VERSION`: current skill version.
- `CHANGELOG.md`: version history.
- `README.md` / `README.zh-CN.md`: English and Chinese documentation.
- `install.mjs` / `install.py`: compatibility wrappers for the root installer.
- `agents/openai.yaml`: OpenAI/Codex display metadata and default prompt.
